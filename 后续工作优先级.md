# “文渊”后续工作优先级

## 1. 当前基线

MVP0～MVP4 均已通过用户验收。当前系统已经具备以下完整核心闭环：

> 注册登录 → 私有知识库 → 文献解析与向量检索 → 分章节报告生成 → 引用追溯 → 相似度检测 → 多风格润色 → 历史版本 → 管理员配置与审计

后续工作的目标不再是证明技术链路可行，而是依次提高原题基本功能覆盖率、体验与运维完善度、部署可靠性、比赛交付质量和加分能力。

总原则：

1. 先补原题基本功能，不把基本功能误归类为可选增强；
2. 基本功能补强后进行体验与运维完善，提升系统易用性与可管理性；
3. 体验与运维完善后进行云端部署，尽早暴露生产环境问题；
4. 云端版本稳定后只选择一至两个容易展示、与核心链路高度相关的加分功能；
5. OCR、翻译、图表等大范围能力在时间不足时主动暂缓。

## 2. P0：原题基本功能补强

**状态**：已完成程序实现、数据库迁移、自动化测试与前端生产构建验证。

P0 完成后，项目在原题基本功能层面应达到接近完整状态。

### 2.1 修复已知问题

- 核查并修复报告组装台中的索引匹配问题；
- 统一前后端时间戳时区、格式化方式和历史记录展示；
- 为上述问题补充可复现样例与回归测试。

**完成标准**：

- 报告章节、引用、版本选择不存在错位；
- 数据库存储 UTC 或带时区时间，前端统一按用户本地时区展示；
- 自动化测试能够覆盖曾出现的问题。

### 2.2 完整的管理员模板管理

- 提供模板列表、新建、编辑、删除和发布；
- 支持章节新增、删除、排序、章节说明和必填输入配置；
- 支持生成字数、检索数量、检索权重等模板参数；
- 新版本发布后，旧报告继续引用创建时的模板版本；
- 对正在被报告使用的模板或版本提供安全删除限制。

**完成标准**：

- 管理员可以不修改数据库或代码完成模板全生命周期管理；
- 发布新版本后，旧报告内容、章节和生成上下文仍可复现；
- 模板关键操作进入审计日志。

### 2.3 内容审核流水线

- 汇总文献与报告中的敏感词命中结果；
- 提供待审核、已通过、已限制、已下架等审核状态；
- 展示命中词、命中位置、所属用户和内容摘要；
- 支持通过、下架、恢复、封禁用户和彻底删除；
- 保存审核人、审核意见、时间和操作结果。

**完成标准**：

- 从敏感词命中到管理员处置形成完整闭环；
- 受限制内容在学生端有明确状态，不能继续被误用；
- 所有处置动作可审计且具备必要的二次确认。

### 2.4 参考文献与引用完整性

- 为文献补充作者、标题、年份、来源等参考文献元数据；
- DOCX 文末按照 GB/T 7714 的受控规则生成参考文献列表；
- 检查“正文有引用但无对应文献”和“参考文献存在但未被引用”；
- 不完整元数据要求用户补充，不允许大模型猜测。

**完成标准**：

- 报告中的引用编号、数据库引用记录和文末参考文献一一对应；
- 缺失或冗余引用能在导出前得到明确警告；
- DOCX 中的参考文献顺序和编号稳定可复现。

### 2.5 校园公告

- 管理员创建、编辑、发布、下线公告；
- 支持置顶、发布时间、下线时间和基础富文本；
- 学生端展示当前有效公告；
- 公告发布、修改和下线进入审计日志。

**完成标准**：

- 管理员可以完成公告完整生命周期管理；
- 过期或下线公告不会继续出现在学生端。

## 3. P1：体验与运维完善

**状态**：已完成程序实现、数据库迁移、自动化测试与前端生产构建验证。邮箱验证码、邮箱验证与找回密码使用一次性验证码、十分钟有效期和单次消费机制，部署时需要配置 SMTP。

P1 不阻塞开发基线完成，可以在 P0 基本功能补强后完成。

1. 个人资料编辑：头像、昵称和基础个人信息；
2. 登录能力：邮箱验证码、找回密码和管理员重置密码；
3. 浅色与深色主题切换；
4. 用户磁盘占用、文献数、报告数和模型调用量统计；
5. 审计日志按时间、操作类型和操作者筛选，并支持导出；
6. 报告历史按日期分组，明确草稿和已完成状态；
7. 单篇文献标签、分类和跨知识库归档；
8. 制作错误码速查表；
9. 将 toast 统一为页面顶部浮动通知；
10. 注册称呼和检索示例文案的动态展示；
11. 继续优化管理员侧边栏折叠与移动端交互。

## 4. P2：云端部署与生产化

完成 P1 体验与运维完善后，执行 P2 云端部署，而不是立即加入 OCR、翻译或图表功能。

### 4.1 容器与服务拆分

- 为前端和 FastAPI 后端编写生产镜像；
- 将文献解析、向量化和报告生成迁移至独立 Worker；
- 使用 Redis + RQ、Dramatiq 或其他受控队列承载长任务；
- 扩展 Docker Compose，至少包含 Web、API、Worker、PostgreSQL 和 Redis；
- 增加健康检查、启动依赖和数据库迁移步骤。

**完成标准**：

- API 进程重启不会导致已入队任务永久丢失；
- Worker 可单独重启，失败任务能够重试；
- 一条命令可以从空环境启动完整系统。

### 4.2 域名、HTTPS 与网络配置

- 配置 Nginx 或 Caddy 反向代理；
- 绑定域名并启用 HTTPS；
- 收紧生产 CORS 和可信主机；
- 设置上传体积、请求超时和 SSE 代理参数；
- 禁止数据库、Redis 等内部端口直接暴露公网。

**完成标准**：

- 公网可以通过 HTTPS 正常访问学生端和管理员端；
- 上传、SSE 报告生成、DOCX 下载和刷新令牌均可正常工作；
- 浏览器控制台不存在生产配置导致的跨域或混合内容错误。

### 4.3 数据、文件与密钥

- 为 PostgreSQL 制定自动备份和恢复演练；
- 本地文件目录使用持久化云盘，或通过现有 `FileStorage` 接口接入 S3/MinIO；
- API Key、JWT Secret 和数据库密码使用服务器密钥或环境变量管理；
- 日志不得包含 Token、API Key 或完整私有文献正文；
- 明确第三方 LLM 数据传输提示和隐私说明。

**完成标准**：

- 服务器或容器重建后，数据库和上传文件可以恢复；
- 生产密钥不进入 Git 仓库、镜像层或前端构建产物；
- 完成至少一次备份恢复演练。

### 4.4 生产回归与监控

- 增加登录、跨用户隔离、上传、生成、查重、润色和管理员权限的 API 集成测试；
- 部署后运行 MVP2～MVP4 外部 LLM 回归脚本并保存脱敏产物；
- 监控 API、Worker、数据库、磁盘空间和任务失败率；
- 配置日志轮转和异常告警。

**完成标准**：

- 生产环境核心流程通过完整烟雾测试；
- 禁用用户、越权访问和无效密钥场景均得到预期错误；
- 故障可以通过日志、任务状态和监控定位。

## 5. P3：比赛加分功能

### 5.1 推荐优先：混合检索与 reranker

- 将关键词检索和向量检索进行候选融合；
- 使用 reranker 对有限候选集重排；
- 保留召回分、重排分、最终排序和模型版本；
- 建立固定问答集，对比 Top-K 命中率、排序质量、延迟和费用；
- reranker 必须可关闭，失败时回退到现有向量检索。

选择理由：与现有 RAG 主链路直接相关，容易通过前后对比证明价值，也能体现算法和工程设计。

### 5.2 次优先：OCR 与复杂 PDF

- 识别扫描版 PDF；
- 保留页码、段落和 OCR 置信度；
- 对低置信度内容提示人工核查；
- 暂不承诺复杂表格和公式的完全恢复。

### 5.3 其他增强

- 英文与多语种文献翻译、摘要和入库；
- 实验数据识别、图表生成与报告插图；
- 学术热点词云、学院活跃度和统计大屏；
- 更丰富的报告模板和学院规范包。

## 6. P4：比赛交付材料

交付材料与程序同等重要，应在功能稳定后并行整理。

- 《程序设计说明书》Word/PDF；
- 系统功能模块分解图；
- 文献归档与报告生成全流程图；
- LLM、提示词和学术助手交互流程图；
- 数据库实体关系图和向量数据结构说明；
- 核心功能运行截图与操作指南；
- 自主实现部分说明：分块、检索、模板匹配、引用映射和查重算法；
- 外部 LLM、离线模式和数据隐私边界说明；
- 自动化测试与验收产物索引；
- 十分钟演示流程和故障备用方案；
- 若为双人团队，补充明确的分工和个人完成情况。

## 7. 推荐执行路线

```text
已知问题修复
  → 模板管理
  → 内容审核
  → 国标参考文献与引用校验
  → 公告
  → 体验与运维完善
  → 独立 Worker 与完整容器化
  → 云端 HTTPS 部署
  → 生产回归与备份恢复
  → reranker 或 OCR 二选一
  → 程序设计说明书与答辩材料收口
```

若时间非常有限，最低推荐范围为：

1. 修复已知问题；
2. 完成模板管理、内容审核和参考文献；
3. 完成关键体验与运维完善；
4. 完成可公开访问的 HTTPS 云端部署；
5. 完成程序设计说明书和演示材料；
6. 暂缓 OCR、翻译、图表和热点大屏。
